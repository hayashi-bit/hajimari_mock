開発方針ドキュメント

miiiroの全ての設計判断の「最上位の判断基準」

コアコンセプト

「白紙のノートと、隣に座っている人」

miiiroは、ユーザーにとって以下の存在である：

白紙のノート

何を書いてもいい。構造も正解もない。自分の言葉で、自分のペースで。

隣に座っている人

指示はしない。でも聞いている。覚えている。たまに「それ、どういうこと？」と聞いてくる。

このメタファーが意味するのは、miiiroは「ツール」ではなく「場」である、ということ。
ユーザーが自分の思考を言語化し、整理し、前に進むための「安全な場所」を提供する。

迷ったときは常にここに立ち返り、「この判断はコアメタファーを強化するか、希薄にするか？」で決定する。

メタファーの2層構造

miiiroのターゲット層は「GPTを開いても何を聞けばいいかわからない」人たちである。「白紙のノート」を渡すだけでは、GPTの空欄と同じ問題が発生する。

この矛盾を解決するために、メタファーは2つの別レイヤーで機能する。

メタファーUI（見た目）AI（振る舞い）
白紙のノート 余計なボタン・カード・装飾がない。入力欄は常に白紙。静か。 プロンプト設計
隣に座っている人 — 話しかけてくるのはこっち。会話の起点はAI側にある。

ノートは白紙のまま。だが隣の人が「今日どうだった？」と声をかけてくるから、ユーザーはそのノートに「答え」を書き始める。

  • 重要隣の人の声が聞こえる（チャット画面にAIのメッセージが先に表示されている）
  • 重要入力欄（ノート）自体は常に白紙である

GPTとmiiiroの構造的な違い

GPTmiiiro
会話の起点ユーザー（能動的行動が必要）AI（ユーザーは受動的でOK）
入力のハードル「何を聞けばいいかわからない」「答えるだけ」
空欄の意味自分で考えて書く場所AIの問いかけに返事を書く場所
使わない日罪悪感・離脱AIが待っていてくれる安心感

具体的な体験設計

  • 起動時アプリを開いた瞬間、AIからメッセージが来ている。「おはようございます。昨日のインスタ投稿、反応どうでしたか？」
  • 深掘りユーザーが「まだ何も進んでない…」と言うと、「忙しかったんですね。ちなみに、何が一番引っかかってます？時間？それとも方向性？」と返す
  • 沈黙OK何も答えなくても、翌日また別の切り口で話しかけてくる。「使わなきゃ」というプレッシャーがない

miiiro サービス定義

自分の意志でアクセルを踏もうとする女性のキャリアを、「方向づけ・アクセル・ブースト」で形にするサービス。

チャットUIデザイン

「凛とした温もり・余白と意志・静かな推進力」

カラーパレット

#C1694F — ユーザーバブル・音声ボタン・CTA #0F3752 — 送信ボタン・ロゴ・見出し #F8F5F0 — AIバブル・セクション背景

核心的なデザイン判断

  • アバター「隣の人」は姿が見えない。アバター画像なし
  • 時刻タイムスタンプ非表示。時間を気にさせない
  • 入力欄プレースホルダーは最小限
  • 音声音声ボタン 48px（主役）。スキマ時間に声で使う
  • 送信送信ボタン 36px（サブ）。テキスト入力は補助
  • 機能引き算の設計。ボタンを増やさない

アニメーション仕様

  • メッセージopacity 0→1 + translateY 8px→0 / 300ms ease-out
  • タイピング3ドットが0.3s遅延で交互にフェード
  • 音声ボタンscale 1→1.08→1 / 1.5s infinite（録音中）
  • 切り替えscale 0.8→1 + opacity / 200ms

UI設計原則

「邪魔しないこと」が最高のUI

やらないこと（禁止事項）

  • 返答カード・選択肢の提示
  • 過剰な装飾・アニメーション
  • 毎回同じ定型挨拶
  • 長文での返答
  • 機能ボタンの乱立

やること（設計方針）

  • 入力欄は「ただの空白」
  • AIの返答はプレーンテキスト
  • 開いた瞬間に使える
  • 音声入力を「主役」にする
  • 成果物は「静かに現れる」

音声入力について

ターゲット層（女性フリーランス）は家事の合間や移動中に使うことが多い。「話しかけるだけで完結する」体験を実現するため、音声入力ボタンはテキスト入力欄と同等以上の存在感で配置する。

AI応答設計

「飽きない」を作るのはUIではなくAIの振る舞い

飽きないかどうかを決める5つの原則

  • 聞く人遮らない。否定しない。まず受け止める。短い相槌を打ってから、1つだけ質問する
  • 覚える人「前に言ってた〇〇の件、どうなりました？」— 事業カルテと会話履歴をDBに保存し、毎回AIに「思い出させる」
  • 意外な問いたまに「それ、本当にやりたいですか？」「逆に、やめたらどうなりますか？」とユーザーが予期しない角度から問いかける
  • テンション元気な日は軽く返す。疲れている日は静かに寄り添う。ユーザーの文体・文量・絵文字の有無からテンションを読み取る
  • 短く返す1回の応答は2〜3文。そして必ず1つだけ質問で終わる

やってはいけないこと

  • 毎回同じ挨拶
  • 選択肢を出す
  • 長文で返す
  • 褒めすぎる
  • 毎回同じトーン

やるべきこと

  • 前回の話題に触れてから始める
  • 「最近どうですか？」とだけ聞く
  • 短く返して、1つだけ質問する
  • たまに率直に「それ、本当にやりたいですか？」と聞く
  • 相手のテンションに合わせる

壁打ちプロンプト設計（3層構造）

Layer 1: 基本ルール（固定）
  • インタビュアーとして振る舞う（コーチ・先生・アドバイザーではない）
  • まず共感 → 感想 → 質問（質問は0〜1個）
  • 2〜3文以内。長文禁止
  • アドバイス・提案・解決策の提示は禁止
  • ユーザーの言葉をそのまま拾う（きれいに言い換えすぎない）
  • 柔らかい語尾（「〜かな？」「〜だったりする？」）
Layer 2: モード別質問方向（動的注入）
  • 強み整理: 人からよく頼まれること、自然にできていることを掘る
  • 悩み整理: 何に引っかかっているか、止まっている原因を掘る
  • サービス整理: 誰のどんな困りごとを助けたいかを掘る
  • ターゲット整理: 一番助けたい人はどんな人かを掘る
  • 意思決定: どの選択肢で迷っているか、本当はどちらに傾いているかを掘る
  • 行動整理: 最初にできそうなこと、10分でできることを掘る
Layer 3: 記憶コンテキスト（動的注入）
  • 事業カルテ（オンボーディングで収集済み）
  • 前回セッションの記憶（重要ワード・強み・迷い・次回テーマ）
  • 直近の会話履歴（最新20件）

4往復ごとにプロンプトに整理指示を動的注入。「ここまで聞いてて思ったんだけど、〇〇ってことだよね？」形式。

会話の最初の2〜3往復後、LLMに別途判定させる。判定結果をセッションに保存し、以降のプロンプトに注入。

まとめ生成スキーマ（LLM構造化出力）

"theme": "今日話したテーマ", "insights": ["見えてきたこと"], "keywords": ["大事そうな言葉"], "strengths": ["見えてきた強み"], "uncertainties": ["まだ迷っていること"], "nextTheme": "次に考えるとよいこと", "nextAction": "小さな次アクション"

技術アーキテクチャ

OpenAI API ≠ ChatGPT — この区別が全ての基盤

miiiroが使うのはOpenAI API（GPT-4o-mini等）であり、ChatGPTという製品ではない。 APIは「頭脳だけ借りる」部品であり、ユーザーのことを一切覚えない。 これは制約ではなく、むしろ強みである。

データフロー

  • ①システムプロンプト（AIの人格・ルール）
  • ②事業カルテ（名前、事業内容、目標、悩み）← DBから取得
  • ③直近の会話履歴（過去5〜10往復）← DBから取得
  • ④今回のユーザーメッセージ → OpenAI APIに送信 → 応答をDBに保存 + ユーザーに表示

つまり「覚えている」のはAIではなく、miiiroのデータベースである。AIには毎回「この人はこういう人で、前回こういう話をしました」と教えてあげる構造。

DBスキーマ

chat_sessions: id, userId, deviceId, title, mode, messageCount, keywords, strengths, uncertainties, nextTheme, summary, summaryGeneratedAt, isFavorite, createdAt, updatedAt messages: id, sessionId, role, content, createdAt business_profiles: id, userId, deviceId, displayName, businessType, targetCustomer, currentGoal, onboardingCompleted, createdAt, updatedAt

なぜこの技術判断をしたか

  • /chat専用ページを分けると「チャットと壁打ちの違いは何？」という説明コストが発生する。ターゲット層にとって選択肢が増えること自体が障壁。既存のDB・UI・音声入力はそのまま活用できる
  • 匿名認証モニターテスト段階でOAuth登録を強制すると離脱率が上がる。デバイスIDベースなら「URLを開くだけ」で壁打ち開始可能
  • LLM判定ターゲット層は自分の状態を正確に言語化できないことが多い。キーワードマッチでは拾えない。LLM判定はコスト微小で精度が圧倒的に高い
  • ボタン生成自動生成にすると毎回LLMコストがかかる。ボタン表示は自動（5往復以上で表示）、生成はタップ時のみ実行
  • 共感ファースト初期プロンプトは「受け止め→即質問」のパターンが繰り返され、問い詰められている感覚になった。実際のモニターフィードバックに基づき修正

記憶の設計

「何を覚え、何を忘れるか」を設計する

  • 継続性無料プランでも会話履歴が残る。保存件数に制限をかけることでプラン差別化が可能
  • マルチデバイス端末を変えても同じ体験。データはクラウドに保存されるため、スマホでもPCでも同一体験
  • 人格制御AIの「人格」を完全にコントロール可能。ChatGPTと違い、miiiro独自のメンター像を自由に設計できる
  • 精度設計記憶の精度を設計できる。「何を覚えるか」「いつ忘れるか」をmiiiro側で制御
  • コスト制御会話履歴の送信量を調整することでAPI費用をコントロール

データ保存構造（概念設計）

ユーザーテーブル └─ 事業カルテ（プロフィール情報） └─ 会話セッション一覧 └─ 各セッションのメッセージ履歴 └─ 自動生成された「まとめ」 └─ AIが抽出した重要トピックタグ
記憶はユーザーが望んだ時だけ残す。強制的なプロファイリングはしない。

成果物の設計

「押しつけがましくない」成果物の提示方法

壁打ちが終わった後、AIが自動的に「まとめ」を生成するが、表示方法は「押しつけない」。主語はあくまでユーザーであり、「AIが勝手に作ったもの」ではなく「自分が話したことの整理」として提示する。

  • 表示条件会話が5往復以上続いた場合のみ、画面下部に小さく「まとめを見る」リンクが現れる
  • 展開タップすると「今日の壁打ちまとめ」カードが展開される
  • 内容話したテーマ / 出てきたアイデア / 明日やること（1〜3個）
  • 保存閉じても履歴からいつでも見返せる
重要：まとめは「AIが勝手に作ったもの」ではなく、「自分が話したことの整理」として提示する。主語はあくまでユーザー。

競合特徴miiiroとの違い

「ユーザーが、自分の言葉で、自分のペースで考えを言語化できる場を、静かに提供し続けること。」

UIは邪魔をしない。AIは押しつけない。でも、話し終わった後には「今日も少し前に進めた」という感覚が残る。それがmiiiroの価値である。

判断基準チェックリスト

新しい機能やUI要素を追加する際の確認事項

  • ▷これは「白紙のノート」の静けさを壊さないか？
  • ▷これは「隣に座っている人」の押しつけがましさを生まないか？
  • ▷ユーザーが「自分の言葉で話している」感覚を維持できるか？
  • ▷これがなくても、ユーザーは目的を達成できるか？（YESなら不要）
  • ▷3ヶ月後も飽きずに使い続けられるか？
  • ▷miiiroカラー・フォントに沿っているか？
  • ▷Vercelへのデプロイが30秒以内に完了するか？

開発履歴

プロダクトの変遷と各フェーズで行った意思決定の記録

v1.2.1 — わたしの可視化：強みカード刷新・自己紹介文AI生成（2026-06-05）

強みカードを刷新。複数セッションで同じ強みが言及された場合は出現回数をカウントし、2回以上出現したものを「CORE」ラベルでハイライト表示（ピンク背景・太字）。初出日と出現回数バッジも表示。自己紹介文生成をテンプレート文字列から Claude API によるストリーミング生成に刷新。SNS・営業提案・交流会の3スタイルを選択可能。生成後にコピーボタンを表示。api/intro.ts を新設。

v1.2.0 — AI記憶：pgvector + OpenAI Embedding 実装（2026-06-05）

Supabase に pgvector 拡張を有効化し、session_memories テーブル（1536次元ベクトル）を作成。まとめ生成後に OpenAI text-embedding-3-small でベクトル化して保存する api/embed.ts を新設。チャット送信ごとに match_memories 関数でコサイン類似度 0.75 以上の過去セッションを最大3件取得し、system prompt に注入。これにより AI が過去の会話文脈を意味的に参照できるようになった。OPENAI_API_KEY は Vercel 環境変数として設定（コード非公開）。

v1.1.3 — 全ページ共通フッターフィードバック・ナビ整備（2026-06-04）

/home にフッターフィードバックフォームを追加（チャットと同仕様）。今後実装する全ページにヘッダー・フッター・フィードバックを共通で実装する方針を確立。ダッシュボード・設計ドキュメントのナビにトップ画面・壁打ちチャットリンクを追加。チャット画面のナビにトップリンクを追加。

v1.1.1 — /me ページ刷新・タイトル修正・5クリック開発モード（2026-06-04）

/me ページを「わたしの可視化」として全面刷新。先週vs今週の継続記録比較・強みカードリスト（番号付き）・思考・行動の傾向4項目（時間帯/曜日/会話量/ペース）・自己紹介文生成ボタンを追加。「hajimari」残存テキストを「miiiro」に修正。/home・/me 両ページのロゴを5回クリックすると開発モードトーストを8秒間表示（ダッシュボード/設計ドキュメントへのリンク）。

v1.1.0 — トップ画面・わたしの状態ページ新設（2026-06-04）

/home：機能ランチャー画面。壁打ちチャットのみ有効で、SNS・タスク管理・議事録・メールは準備中（SOON）表示。右上の「わたしの状態」ボタンで/meへ遷移。時間帯に応じた挨拶文を表示。

/me：わたしの状態ページ。localStorageのセッションデータを集計して連続利用日数・今週の会話数・累計会話数・強みタグ・時間帯×曜日ヒートマップ・深夜アラート・AIメッセージを表示。サービス名をmiiiroに統一。

v1.0.2 — AI応答設計をロジャーズ式傾聴に刷新（2026-06-04）

「毎回必ず1質問」というルールを廃止し、カール・ロジャーズの傾聴技法（反映・ミラーリング）を中心に据えたプロンプト設計に刷新。相手の言葉をそのまま使って返す「反映」を最優先とし、質問は3〜4往復に1回以下に抑える。テンション自動読みも強化し、短文メッセージには相槌のみ・長文には深掘り1つという明確な基準を設定。モード別プロンプトも全面的に反映スタイルへ更新。「沈黙・余白を作る」概念を明示的に導入。

v1.0.1 — モード選択UI改善・まとめをチャット内表示（2026-06-04）

モード選択UIをフルスクリーンオーバーレイからチャット上部の小バーに変更。選択しなくてもそのまま会話開始可能（デフォルト動作は傾聴相当）。まとめ促進のアコーディオンバーを廃止し、8往復後にAIのバブルとしてチャット内に「ここまでをまとめる」ボタンを表示。まとめ結果もその場に表示される。

v1.0.0 — 会話モード選択機能（2026-06-04）

チャット開始時に「壁打ち／傾聴／雑談」の3モードを選択するオーバーレイを実装。モード別にシステムプロンプトを切り替えてAIの振る舞いを制御。会話中は「モードを変える」ボタンでいつでも変更可能。またユーザーのメッセージ量・テンションを自動読みして質問頻度を動的に調整するテンション自動読みも追加。

v0.9.2 — 天気機能・居住地プロフィール追加（2026-06-04）

プロフィールに居住地フィールドを追加。Open-Meteo API（無料・APIキー不要）でジオコーディング→天気取得を実装。居住地設定済みなら起動時に自動で天気を取得しシステムプロンプトに注入。未設定時はAIが会話の中で自然に居住地を聞く。

v0.9.1 — フィードバックフォーム・履歴ボタン改善（2026-06-04）

フィードバックフォームに✕ボタンを追加し、フォーム外クリックで自動的に閉じるよう改善。履歴の★・×ボタンをホバーなしで常時表示。現在時刻（JST）をシステムプロンプトに注入し、深夜にランチを聞くなど時間帯がおかしい問題を修正。

v0.9.0 — AI記憶機能・UX大幅改善（2026-06-04）

過去セッションのまとめ（summary）をlocalStorageから収集し、新しい会話のシステムプロンプトに「この人との過去の会話から見えてきたこと」として注入。AIが会話の傾向・癖・課題感を蓄積していく設計。匿名ユーザーID（hajimari_uid）をlocalStorageに生成し、将来のpgvector連携に備える。

v0.7.0 — チャット体験の大幅改善（2026-06-04）

磯崎（大沼）亜樹さんのフィードバックを受け、チャット体験を全面的に改善。課題1〜4をまとめて実装した。

【課題1: コンテキスト不足】プロフィール登録ページ（/profile）を新設。職種・所属・業務内容・チーム構成・課題・目標・キーワードの8項目をlocalStorageに保存し、チャット開始時にシステムプロンプトとして注入。初回から「あなたのことを知っている」状態で会話を始められる。

【課題2: 質問がフワフワ】システムプロンプトを3フェーズ構造（解決したいこと→現状→理想）に再設計。共感を先に置き、漠然とした答えには「どんなとき？」「たとえば？」で自然に深掘り。責められている感を出さない聞き方を意識した設計。

【課題3: 会話が突然終わる】進捗バー＋フェーズラベルを画面上部に表示。8往復後に「ここまでをまとめる」ボタンが出現し、押すと「解決したいこと・現状・理想・見えてきた強み」の構造化サマリーカードをその場に生成。

【課題4: テキストだけでは業務説明が困難】📎ファイル添付ボタンを実装。txt/md/csvファイルの内容をAIのシステムプロンプトに追加し、資料を踏まえた会話が可能に。PDFはフォールバックメッセージで案内。
