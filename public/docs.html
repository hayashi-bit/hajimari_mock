開発方針ドキュメント

hajimari — 設計思想とルールの全体像

コアコンセプト

hajimariは「壁打ち相手としてのAI」です。コーチでも、アドバイザーでも、検索エンジンでもない。 ただ話を聞いて、問いかける存在。

hajimariが提供するのは答えではなく、問い。
ユーザーが自分の中にすでに持っている答えを、引き出す触媒になる。

ターゲット

女性フリーランス。一人で仕事をしていて、決断を一人で抱えている人。

フリーランス 女性 孤独な意思決定 壁打ちニーズ

UI設計原則

カラーシステム

#F87C62 — Salmon Orange（行動・温もり） #0F3752 — Navy（信頼・深さ） #FFFBF5 — Warm White（余白・静けさ）

タイポグラフィ

  • 見出しDM Serif Display — 品格と余白
  • 本文System UI / Noto Sans JP — 読みやすさ優先

レイアウト原則

  • 01余白を恐れない。詰め込まない
  • 02モバイルファースト。100dvhで画面を使い切る
  • 03インタラクションはサブトル（微細）に。主役は会話

AI応答設計

hajimariのAIは以下のルールで動作します。

  • 01返答は2〜3文以内
  • 02最後に質問を1つだけする
  • 03相手の言葉を否定せず、まず受け止める
  • 04選択肢は出さない（ユーザーに考えてもらう）
  • 05相手のテンション・文量に合わせてトーンを調整する
  • 06定型挨拶は使わない
  • 073回に1回は予想外の角度から質問する
AIは先に話しかける。ユーザーを「空白」の前に立たせない。
最初のメッセージはAIが投げる。これがhajimariの最重要原則。

技術アーキテクチャ

設計方針

「シンプルさは信頼性の別名」。フレームワーク・ライブラリを最小化し、 将来の変更コストを下げる。

  • FrontendPlain HTML + CSS + Vanilla JS（ビルドステップなし）
  • APIVercel Serverless Functions（TypeScript）
  • AIAnthropic Messages API（SSEストリーミング）
  • HostingVercel（GitプッシュでCI/CD自動化）

ストリーミングの仕組み

Anthropic APIの content_block_delta イベントをリアルタイムにブラウザへ転送。 体感速度を最大化し、「AIが考えている」感を演出する。

記憶の設計

現時点（Sprint 3）では会話履歴はブラウザのメモリ内のみ。ページをリロードするとリセットされる。

Sprint 4 以降の方向性

  • 短期セッション内の文脈はすべてAPIに渡す（現行）
  • 中期Supabase等で会話ログを永続化
  • 長期感情タグ・テーマ抽出による「振り返り機能」
記憶はユーザーが望んだ時だけ残す。強制的なプロファイリングはしない。

成果物の設計

hajimariは「答え」を返さない。ユーザーの思考の解像度を上げることが成果物。

価値の測定指標

  • ◎「それは考えてなかった」という発言が生まれた
  • ◎会話後に「やること」が1つでも明確になった
  • ◎翌日もまた話しかけたくなった
  • ×「答えを教えてくれた」だけで終わった

判断基準チェックリスト

機能追加・変更時は以下を確認する。

  • ▷この変更は「壁打ちの質」を上げるか？
  • ▷コードの複雑度が増すか？増すなら本当に必要か？
  • ▷モバイルで違和感なく使えるか？
  • ▷ユーザーを迷わせる要素はないか？
  • ▷AIが「答え」を出していないか（出すべきでない局面で）
  • ▷hajimariカラー・フォントに沿っているか？
  • ▷Vercelへのデプロイが30秒以内に完了するか？
