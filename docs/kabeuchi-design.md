# miiiro 壁打ち機能「あるべき姿」整理

## 1. 前提：GPTとの壁打ち内容から何が見えたか

GPTとの議論で出た設計方針は筋が通っている。ただし、そのまま実装仕様にするには粒度が粗い箇所と、現在の実装との関係が未整理な箇所がある。以下、miiiroの既存コードベースに照らして「何をどう変えるべきか」を整理する。

---

## 2. 結論：/chat を壁打ち専用に作り替える（別ページにしない）

**推奨：現在の /chat をそのまま壁打ち機能に進化させる。別ページ（/wall 等）は作らない。**

理由は3つ。

第一に、現在の /chat は「AIメンターとの対話」として設計されており、壁打ちの上位互換にできる。オンボーディング → 通常会話 → 壁打ちという流れは同じ画面内で自然に遷移できる。ページを分けると「チャットと壁打ちの違いは何？」という説明コストが発生する。ターゲット層にとって、選択肢が増えること自体が障壁になる。

第二に、現在のDBスキーマ（chatSessions + messages）は壁打ちセッションの基盤としてそのまま使える。セッションにフィールドを追加するだけで壁打ち固有のデータ（モード、記憶、まとめ）を持てる。

第三に、UIコンポーネント（Chat.tsx）は既にスマホ最適化済みで、音声入力も実装済み。壁打ち用の追加UI（まとめ表示、モード表示）はこの上に載せるだけでよい。

---

## 3. 現在の /chat と壁打ちの差分

現在の実装と、壁打ちとして「あるべき姿」の差分を整理する。

| 項目 | 現在の /chat | 壁打ちとしてあるべき姿 |
|---|---|---|
| **システムプロンプト** | AIメンター（汎用対話） | インタビュアー（質問で深掘り、アドバイス禁止） |
| **会話の主導権** | 曖昧（AI側だが指示が弱い） | 明確にAI主導。1問ずつ質問、ユーザーは答えるだけ |
| **モード判定** | なし | 6モード自動判定（強み・悩み・サービス・ターゲット・意思決定・行動） |
| **途中整理** | なし | 3〜5往復ごとにAIが短く整理し、合っているか確認 |
| **まとめ生成** | なし | 5往復以上で「今日のまとめを見る」を表示 |
| **記憶保存** | 会話履歴のみ | 重要ワード・見えてきた強み・迷い・次回テーマをDB保存 |
| **次回起動** | 汎用挨拶を生成 | 前回の記憶から具体的な問いかけで再開 |
| **オンボーディング** | 事業カルテ収集（3問） | そのまま維持。完了後に壁打ちモードへ移行 |

## 4. システムプロンプトの再設計

現在のプロンプトは「AIメンター」として設計されているが、壁打ちの核心は**インタビュアー**として振る舞うこと。ここが最大の変更点。

### 現在のプロンプトの問題

現在の `MIIIRO_SYSTEM_PROMPT` は「隣に座っている人」のトーンは良いが、会話の進行ルールが弱い。「1つだけ質問で終わる」とは書いてあるが、**何のために質問するのか**（深掘りの方向性）が定義されていない。結果として、汎用的な雑談AIになりやすい。

### あるべきプロンプト構造

壁打ち用プロンプトは以下の3層で構成すべき。

**Layer 1: 基本ルール（固定）**
- インタビュアーとして振る舞う。コーチ・先生・アドバイザーではない
- 1回に質問は1つだけ
- まず受け止める → 短く整理 → 次の質問
- 2〜3文以内。長文禁止
- アドバイス・提案・解決策の提示は禁止
- ユーザーの言葉をそのまま拾う（きれいに言い換えすぎない）

**Layer 2: モード別の質問方向（動的注入）**
- 現在のモード（強み整理・悩み整理等）に応じた質問の方向性を注入
- モードはAIが会話から自動判定し、プロンプトに含める

**Layer 3: 記憶コンテキスト（動的注入）**
- 事業カルテ（既存）
- 前回セッションの記憶（重要ワード・強み・迷い・次回テーマ）
- 直近の会話履歴

### オンボーディングとの関係

現在のオンボーディング（事業カルテ収集）はそのまま維持する。オンボーディング完了後、最初の壁打ちセッションが始まる。オンボーディングで集めた情報はLayer 3の記憶コンテキストとして壁打ちプロンプトに渡す。

## 5. 6つのインタビューモード

GPTとの議論では7モード（発信方向性整理モードを含む）が挙がっていたが、**6モードに絞る**。「発信方向性整理」はSNS投稿生成と密結合しやすく、今回のスコープ外。

| モード | 目的 | 質問の方向 |
|---|---|---|
| **強み整理** | 得意なこと・頼まれやすいことを言語化 | 「人からよく頼まれること」「自然にできていること」を掘る |
| **悩み整理** | 引っかかっていること・止まっている理由を整理 | 「何に引っかかっているか」「止まっている原因は何か」を掘る |
| **サービス整理** | 提供したい価値・サービス内容を整理 | 「誰のどんな困りごとを助けたいか」を掘る |
| **ターゲット整理** | 誰に届けたいかを具体化 | 「一番助けたい人はどんな人か」を掘る |
| **意思決定** | 迷っている選択肢を整理 | 「どの選択肢で迷っているか」「本当はどちらに傾いているか」を掘る |
| **行動整理** | 考えた内容を次の小さな行動に落とす | 「最初にできそうなこと」「10分でできることは何か」を掘る |

### モード判定のロジック

ユーザーにモードを選ばせない。AIが会話の最初の数往復から自動判定する。

実装方法は2つ考えられる。

**案A: LLMに判定させる（推奨）**
- 会話の最初の2〜3往復後、LLMに別途「この会話のモードは何か」を判定させる
- 判定結果をセッションに保存し、以降のプロンプトに注入
- 会話の途中でモードが変わった場合も再判定

**案B: ルールベースでキーワード判定**
- 「強み」「得意」→ 強み整理、「迷っている」「決められない」→ 意思決定、等
- 精度が低い。ターゲット層は自分の状態を正確に言語化できないことが多いため、キーワードマッチでは拾えない

案Aを推奨する。LLM呼び出しが1回増えるが、判定精度が圧倒的に高い。コストは微小（判定用プロンプトは短い）。

## 6. 途中整理

3〜5往復ごとに、AIが短く整理する。これはシステムプロンプトで制御する。

### 実装方針

途中整理のタイミングは、往復数をサーバー側でカウントし、プロンプトに「そろそろ短く整理してください」という指示を動的に注入する。

整理の形式は以下の通り。

> 「ここまで聞くと、〇〇というより、△△が得意そうです。この表現は近いですか？」

ポイントは2つ。
- 整理した後、必ず「合っているか」を確認する質問で終わる
- 整理は1〜2文。箇条書き禁止

---

## 7. まとめ生成

### トリガー条件

- ユーザーの発言が5回以上
- ユーザーが「まとめて」と発言
- AIが十分に整理できたと判断（これはプロンプトで制御）

### UI表示

会話エリアの下部に「今日のまとめを見る」ボタンを小さく表示する。タップでボトムシート（またはモーダル）が開く。会話の流れを邪魔しない。

### まとめの内容（LLMに構造化JSONで生成させる）

```
{
  "theme": "今日話したテーマ",
  "insights": ["見えてきたこと1", "見えてきたこと2"],
  "keywords": ["大事そうな言葉1", "大事そうな言葉2"],
  "uncertainties": ["まだ迷っていること"],
  "nextThoughts": ["次に考えるとよいこと"],
  "nextActions": ["小さな次アクション"]
}
```

このJSONはDBに保存し、次回セッション開始時のプロンプトに渡す。

## 8. 記憶の設計

### DBスキーマ変更

現在の `chatSessions` テーブルに以下のカラムを追加する。

| カラム | 型 | 説明 |
|---|---|---|
| `mode` | varchar(50) | 現在のインタビューモード（strength/worry/service/target/decision/action） |
| `keywords` | text (JSON) | 抽出された重要ワード |
| `strengths` | text (JSON) | 見えてきた強み |
| `uncertainties` | text (JSON) | 迷っていること |
| `nextTheme` | text | 次回聞くべきこと |
| `summary` | text (JSON) | 生成されたまとめ（上記JSON構造） |
| `summaryGeneratedAt` | timestamp | まとめ生成日時 |

### 記憶の更新タイミング

- **モード**: 会話の最初の2〜3往復後にLLMで判定し保存。途中で変わった場合も更新
- **keywords / strengths / uncertainties**: まとめ生成時にLLMで抽出し保存
- **nextTheme**: まとめ生成時にLLMで生成し保存
- **summary**: まとめ生成時に保存

### 次回起動時の流れ

1. 前回セッションの `summary` と `nextTheme` を取得
2. 新しいセッションを作成
3. 挨拶プロンプトに前回の記憶を注入
4. AIが前回の続きから問いかける

例：「前回、"人の考えを整理するのが得意かも"という話が出ました。今日は、その強みを誰に届けるかを少し考えてみますか？」

## 9. 実装の優先順位

MVP（最小限の動くもの）として、以下の順で実装する。

| 優先度 | 項目 | 工数感 | 理由 |
|---|---|---|---|
| **P0** | システムプロンプト書き換え（インタビュアー化） | 小 | これだけで体験が劇的に変わる。コード変更は chat.ts のプロンプト文字列のみ |
| **P0** | 途中整理ロジック（往復数カウント → プロンプト注入） | 小 | サーバー側で往復数を見て条件分岐するだけ |
| **P1** | モード自動判定（LLM判定 → セッション保存） | 中 | DBスキーマ変更 + LLM呼び出し追加 |
| **P1** | まとめ生成（LLM構造化出力 → DB保存 → UI表示） | 中 | 新しいtRPCエンドポイント + UIコンポーネント追加 |
| **P1** | 記憶保存（keywords/strengths/uncertainties/nextTheme） | 中 | まとめ生成と同時に実装 |
| **P2** | 次回起動時の記憶引き継ぎ | 小 | 既存の getGreeting を改修するだけ |
| **P2** | まとめ表示UI（ボトムシート） | 小 | フロントエンドのみ |

**P0だけで壁打ちの核心体験は成立する。** プロンプトを書き換えて途中整理を入れるだけで、「AIが質問で深掘りし、途中で整理してくれる」体験になる。モード判定やまとめ生成はその上に載せる追加機能。

## 10. 変更しないもの

以下は現状維持。

- **オンボーディングフロー**: 事業カルテ収集は現在の実装のまま。壁打ちの前提情報として有用
- **Chat.tsx のUI構造**: メッセージ表示・入力欄・音声入力は現在のまま。まとめボタンを追加するだけ
- **DBの messages テーブル**: 変更なし。メッセージの保存構造はそのまま
- **/chat-preview**: 静的デモとして残す

## 11. 判断が必要な点

### 11-1. セッションの切り替え

現在は1ユーザー1セッション（最新のセッションを使い回す）。壁打ちでは「テーマごとにセッションを分ける」方が記憶管理しやすい。

**推奨：「新しい壁打ちを始める」ボタンを追加し、テーマごとにセッションを分ける。** 過去のセッション一覧は後回しでよいが、新規セッション作成は必要。

### 11-2. まとめの自動生成 vs 手動トリガー

GPTとの議論では「5往復以上で表示」とあったが、自動生成にするとLLMコストが毎回かかる。

**推奨：ボタン表示は自動（5往復以上で表示）、まとめ生成はボタンタップ時に実行。** これでコストを抑えつつ、ユーザーが見たいときだけ生成する。

### 11-3. 画面名

GPTとの議論では「壁打ちノート」が候補に挙がっていたが、ヘッダーに表示する名前として「壁打ちノート」はやや硬い。

**推奨：ヘッダーは「miiiro」のまま。** 壁打ちは機能名であってページ名ではない。ユーザーにとっては「miiiroを開いたら会話が始まる」だけでよい。

## 12. 現在の /chat からの変更箇所まとめ

### サーバー側（server/routers/chat.ts）
1. `MIIIRO_SYSTEM_PROMPT` を壁打ちインタビュアー用に全面書き換え
2. `buildSystemPrompt` にモード別質問方向・記憶コンテキストの注入を追加
3. `sendMessage` に往復数カウント → 途中整理指示の動的注入を追加
4. 新エンドポイント: `generateSummary`（まとめ生成）
5. 新エンドポイント: `detectMode`（モード自動判定）— sendMessage内で内部的に呼ぶ形でもよい

### DB（drizzle/schema.ts）
1. `chatSessions` に mode, keywords, strengths, uncertainties, nextTheme, summary, summaryGeneratedAt を追加

### フロントエンド（client/src/pages/Chat.tsx）
1. 「今日のまとめを見る」ボタン（5往復以上で表示）
2. まとめ表示コンポーネント（ボトムシート or モーダル）
3. 「新しい壁打ちを始める」ボタン
