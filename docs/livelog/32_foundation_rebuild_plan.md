# 足場（環境）再構築 計画 — 有料リリース前の根本解決

作成: 2026-06-22 ／ 目的＝「小さいことのたびに失敗/エラー連発」の根本（足場の不整合・不可視）を、
有料リリース前に一度きり直す。**データ・モニター・ドメインは失わない。**
関連: doc31（方法B）／#memolog DBドリフト4回。

## 1. なぜやるか（根本問題）
個別エラーを潰し続けるのが非効率の正体。発生源は機能でなく足場：
1. **観測性ゼロ**：本番ログが見えない＝Manusが本番を目隠しデバッグ→診断に時間がかかる。
2. **DB/接続の不整合**：dev/本番のDBがズレる・SQLツールが本番デフォルト→事故（ドリフト4回）。
3. **長時間ジョブが不安定**：Autoscaleのコールドスタート/staleでバックフィルが死ぬ。
4. **検証環境が無い**：本番で初めてエラーが出る（モニターの前で）。
5. **コードの一貫性不足＋ad-hocマイグレーション**：触ると隠れた地雷（tRPC/Express混在等）。

## 2. 引き継ぐもの（壊さない）
- **本番データ**（林の参戦履歴・友達・プロフィール・gmail連携）。
- **動いている機能**（記録/友達/Gmail抽出は正しく動き始めている）＝コードは活かす。
- **liverico.app ドメイン**と**モニターのアクセス**（継続利用中）。
→ 「再構築」＝**足場（環境・DB構成・マイグレーション・観測性・デプロイ）を作り直す**。
  アプリやデータを白紙にすることではない。

## 3. 目標の原則（クリーンな足場）
1. **3環境を明確分離**：production / staging / development。各々専用DB＋明確な命名。共有/本番デフォルトのツールを無くす。
2. **観測性**：本番のログ・エラーにアクセスできる（目隠しデバッグをやめる）。
3. **マイグレーション一本道**：schema.ts を正に、ファイル化（versioned）、dev→staging→prodへ順に適用。
4. **長時間ジョブが死なない**：バックフィル等をコールドスタート/staleで殺さない構成（常時起動ワーカー or 分割ジョブ＋resumable）。
5. **デプロイパイプライン**：git → 自動チェック(テスト/スモーク) → staging検証 → 本番へ昇格。

## 4. 分かれ道（2案）＋判断材料
今日の地雷の多くはManusプラットフォーム固有（本番ログ不可視・SQLが本番デフォルト・コールドスタート・URLがセッション依存）。これらが**直せるか/根本的か**で案が決まる。

| 案 | 内容 | 速さ | 根治度 | 主なリスク |
|---|---|---|---|---|
| **X. Manus内クリーン化** | 方法Bを徹底＋Manus上で観測性/ステージング/常時起動を確保 | 速い | プラットフォーム固有制約が残るなら中 | 本番ログ不可視等が「直せない」と残る |
| **Y. 標準スタックへ移行** | コード/データを保持しつつ、本番ログが見える標準ホスト＋標準マイグレーション＋本物のステージングへ | 遅い（移行作業大） | 高（根治） | 移行コスト＋Manus結合度＋「今後だれが実装するか」 |

### 決め手＝Manusに先に確認すべきこと（これで案が決まる）
1. **本番のログ/エラーにアクセスできるようにできるか**（観測性）。
2. SQLツールの本番デフォルトを恒久的に外せるか・接続先を安全に固定できるか。
3. 長時間ジョブ/Heartbeatをコールドスタートで殺さず安定実行できるか（Reserved等）。
4. 固定URLの**本物のステージング**を用意できるか。
→ これらが**Manus上で確保できる＝案X（速い・在宅で根治）**。
　 **本番ログ不可視等が根本的に無理＝案Y（移行で根治）**を検討。

## 5. 「今後だれが実装するか」も論点
Manusの価値は自動実装の速さ。案Y（脱Manus）は観測性を得る代わりに、**新機能を誰が作るか**が変わる
（Manusが使えなくなる/クロコ＋人手が増える）。＝技術だけでなく開発体制の判断。案Xならこの問題は出ない。

## 6. 安全な移行の原則（big-bang禁止）
- **並行構築**：新しいクリーンな足場を別に立て、現本番は動かしたまま。
- **データ移行は検証付き**：件数・整合をチェックしてから切替。
- **切替はロールバック可能に**：問題が出たら戻せる。
- **モニターを止めない**：liverico.app は維持、切替はダウンタイム最小。

## 7. 次の一歩
1. 上記「4.決め手」をManusに質問し、プラットフォーム制約が**直せるか/根本的か**を確定。
2. その回答で案X/Yを決める。
3. クロコが、決まった案の**具体的な目標構成図＋データを失わない移行手順**を起こす。
