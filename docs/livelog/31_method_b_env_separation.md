# 方法B：環境整理（dev/staging/prod 分離＋マイグレーション一本道）Manus指示

作成: 2026-06-22 ／ 目的＝「どのDB？スキーマ食い違い」が4回再発した根本を断つ。
関連: #memolog 再発防止メモ（DEV/PRODドリフト4回）／doc28 Gmail仕上げ。

## なぜやるか（背景）
同型のDBドリフトが4回発生し、そのたびに1日級の時間を失っている。原因は機能ではなく**環境の足場**：
- アプリ(dev)＝`DATABASE_URL_DEV`（TiDB ap-northeast-1 `/test`）
- 本番アプリ＝`DATABASE_URL`（TiDB us-east-1 `/nHdBwH2e…`）
- SQLツール `webdev_execute_sql`＝`DRIZZLE_DATABASE_URL`（**us-east-1＝本番DB**）←地雷
- マイグレーションが手動 `ALTER TABLE` のアドホック運用＝片側だけ適用→ドリフト

→ Gmail・課金・admin… 何を触っても同じ穴を踏む。**一度ここを整理すれば、以降の全作業が速くなる。**

## ゴール
1. 環境（本番／検証／開発）と、それぞれが使うDBが**一目で明確**。
2. **SQL/マイグレーションツールの既定接続先が“本番”ではない**（事故防止）。
3. スキーマ変更は**マイグレーションファイル（versioned）で一本道**に流す（手アドホック禁止）。
4. **本番を絶対汚さずに検証できる**（独立した検証環境）。
5. **ドリフト検知**（デプロイ前に dev/検証/本番のスキーマ一致を確認）。

## ⚠️ リスク階層と進め方（重要・沼化を防ぐ）
方法B自体を雑にやると「テスト→エラー発見→再構築」の新たな沼になりうる（特にBの
マイグレーション基盤導入は、既存DBとのズレが噴き出す）。そこで**実行は安全な所だけ／
仕組みの作り替えは提案だけ**にして、危険な判断は林が起きている時に行う。

- **P1（実行可・低リスク・アプリ挙動を変えない・可逆/読み取り優先）：A・C・E**
  → 不在中に走らせても新たな沼を作らない。最大の地雷（SQLツール既定=本番）をここで外す。
- **P2（今夜は実行しない＝設計“提案だけ”・朝に林が判断）：B・D**
  → B=マイグレーション基盤の本格導入、D=独立ステージング（新インフラ/コスト）。
  既存スキーマとのズレや費用が絡むため、手順・リスク・コストを提案として出すに留め、実行しない。

**大原則：** アプリの実行挙動を変えない。可逆・読み取り優先。「大改修になりそう／既存スキーマとの
ズレが噴き出す」と感じたら、実行せず提案にまわす。

## 必須要件

### A.【P1・実行可】接続先の棚卸しと是正（最優先・新インフラ不要）
- DB関連の環境変数を全部、表で文書化：変数名 → ホスト/DB名 → 用途 → 対象環境（本番/検証/開発）。
- **SQL/マイグレーションツールの既定接続先を本番から外す**（開発/検証DBに向ける）。本番に対して実行する場合は、明示的・意図的な選択を必須にする（うっかり本番ALTERを物理的に防ぐ）。
- 現状の誤り（`DRIZZLE_DATABASE_URL`が本番を向く）を是正。

### B.【P2・今夜は提案のみ】マイグレーション一本道（既存DBとのズレが出るため実行は朝に判断）
- `drizzle/schema.ts` を唯一の正（source of truth）にする。
- **drizzle-kit でマイグレーションファイルを生成し、リポジトリに versioned で残す**（`drizzle/migrations/` 等）。手書きの ad-hoc `ALTER TABLE` をやめる。
- 適用は順番固定：**開発/検証DBに適用 → 確認 → 本番DBに適用**。各環境への適用コマンドを README 化。
- 1つのマイグレーションは**全環境に適用**（片側だけにしない）。

### C.【P1・実行可】ドリフト検知（読み取り専用・新インフラ不要）
- デプロイ前に **dev/検証/本番のスキーマが一致しているか**を確認する手段を用意（drizzle-kit の check、または各DBの information_schema を比較する小スクリプト）。差分が出たら止める。

### D.【P2・今夜は提案のみ】独立した検証環境（新インフラ/コスト＝林判断）
- Gmail E2E等の検証を、**本番とは別の固定URL＋別DB**で行えるようにする（「本番のenvを一時切替→戻す」運用をやめる）。
- これに新しいTiDBインスタンスやデプロイURL（=コスト/林の判断）が必要なら、**着手前に選択肢とコストを提示して林に確認**。不要（既存の dev DB/URL を検証用に固定運用できる）なら、その構成で進める。

### E.【P1・実行可】残骸・不整合の是正（範囲限定・バックアップ記録のうえで）
- 既存のテスト残骸（他人Gmail連携=yukato等・幽霊データ）を**正しいDBに対して**掃除（1ユーザー=自分のGmail1本）。
- gmailテーブル群が dev/本番で一致しているか最終確認（先日 gmail_sync_results 等は揃えた、その状態を維持）。

## 非交渉（安全）
- 本番DBに対する破壊的SQL（DROP等）は、**明示確認＋事前バックアップ/記録**なしに実行しない。
- 検証データは検証/開発環境に。**本番に流さない。**
- 既存の本番データを失わない。

## 止まって確認すべき時
- P2（B・D）は今夜は実行しない＝提案を出して止まる（朝に林が判断）。
- 本番に対する破壊的変更が必要な時。
- 「大改修になりそう／既存スキーマとのズレが噴き出す」と感じた時は、実行せず提案にまわす。

## 完了報告（朝に見られる形で）
- P1の結果：環境マップ表／是正した地雷（SQL既定を本番から外した）／ドリフト差分（読み取りのみ・修正せず）／残骸掃除。
- P2の提案：B（マイグレーション基盤・移行手順とリスク）／D（独立検証環境・構成と概算コスト）。林の判断が要る点を明示。
