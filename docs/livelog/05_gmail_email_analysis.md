関連: `03_gmail_integration_spec.md`（基本仕様）, `04_monetization_strategy.md`（課金）

実ユーザー（参戦ヘビーユーザー）のGmailから書き出した **9 ZIP / 全345通（ユニーク304通）** のチケット系メールを解析した結果と、それを踏まえた実装計画・データ項目仕様をまとめる。

## 0. 結論サマリー

- 大小すべてのライブが対象にできる（COUNTDOWN JAPAN等の大型フェス〜新宿LOFT等の小箱まで実在を確認）。
- 購入実績系 **184通 → 約54公演** の参戦履歴が「手入力ゼロ」で復元できる。
- = 新規ユーザーが初日から「54公演保有のアクティブユーザー」になり、コールドスタート（最初データが空で離脱）を一発で解消できる。これが本機能の最大価値。
- 営業/通知/野球メール（119通）は送信元では弾けない（巨人戦もe+から届く）→ 本文をLLMで判定する必要がある。これがLLM解析（G1-1）必須の根拠。

## 1. 公開方針（決定済み 2026-06-16）

- **テストモードで当面運用**（OAuth同意画面＝テスト公開ステータス・最大100テストユーザー）。
- gmail.readonly は制限付きスコープ。一般公開にはGoogleのCASA含むセキュリティ審査（数週間）が必要なため、当面は審査を回避。
- 7月「有料100人」ゴールと上限が一致 → 審査ブロックなしで即リリース可能。100人超で「フル審査」or「転送アドレス方式」へ移行を再検討。
- 制約: テスト公開ではリフレッシュトークンが**約7日で失効** → 「再連携導線（G2-3）」が必須。

## 2. Gmailアクセスの仕組み（非エンジニア向け説明）

### なぜアプリがメールを読めるのか＝OAuth
1. ユーザーがアプリで「Gmail連携」を押す
2. Google公式画面へ遷移 →「このアプリにメール閲覧を許可しますか？」
3. ユーザーが「許可」を押す
4. Googleがアプリに鍵（トークン）を発行
5. アプリはその鍵でGoogle公式API経由でメールを読む（**読み取り専用 gmail.readonly**・送信/削除不可）

- パスワードは渡さない。ユーザー本人の許可で読める。いつでも連携解除でき、その瞬間に鍵が無効化される。

### 定期的に勝手に読めるか
- 技術的には可能（リフレッシュトークンでユーザー不在でもサーバーが定期取得）。
- ただしテストモードは約7日でトークン失効 → 当面は「週1で再連携」。フル審査クリア後に完全自動化（G3-2）。

## 4. 項目仕様：メールから取れるもの【全仕分け】

取得率は「正規表現での実測値（184通中）」。本文に情報があってもラベル差で取りこぼす項目はLLM化で大きく改善する。

### 🟢 既存項目（現アプリ仕様にある）→ そのまま埋まる

| 項目 | 取得率（正規表現） | 備考 |
|---|---|---|
| ステータス | 100% | 当落〜確保まで全段階 |
| チケット会社 | 100% | 送信元で確定 |
| 受信日時 | 100% | 並び替え用 |
| sourceMessageId（重複防止） | 100% | 同一メール二度登録しない |
| 会場 | 92% | 全角正規化が必要 |
| 公演日時（開場/開演） | 88% | 受信日≠公演日に注意（転送で顕著） |
| 公演名 | 83% | LLMで95%+想定 |
| チケット形態（紙/電子） | 判別可 | 電子/スマチケ/QR=117通, 発券/店頭/Loppi=109通 |
| 券種/席種 | 54% | |
| 金額 | 7% | 本文にはある→LLMで改善 |
| 座席 | 13% | 同上 |
| 締切日・締切時刻 | 22% | 同上 |

### 🟡 仕様にあるが弱い → LLMで強化

- アーティスト/出演者: 単独公演は「公演名＝アーティスト」。フェスは出演者リスト(21通)/対バン(17通)から抽出、または後段で補完。「公演名がアーティストかフェス名か」の判定こそLLMの出番。
- 金額/URL/番号/期限: 本文に存在するが販売元ごとにラベルが違い正規表現が取りこぼす → LLMでほぼ◎に。
- 都道府県（19%）: 本文明記が少ない → 会場名からLLMの地名知識で補完（渋谷FOWS→東京, Zepp Yokohama→神奈川）。

## 7. 確定した新項目（2026-06-16 決定）

184通の調査を踏まえ、現アプリに以下の新項目を追加することを決定。判断軸は「失念防止＝アプリのコア価値に直結するか」「出現数が実装コストに見合うか」。

| 新項目 | フィールド案 | 型 | 採用理由 |
|---|---|---|---|
| 購入/申込締切日 | `applyDeadline` | date | 申込忘れ防止。出現94通 |
| 当落発表日 | `lotteryResultDate` | date | 当落チェック忘れ防止。出現24通 |
| 入金期限 | `paymentDeadline` | datetime | 失念で予約無効を防ぐ。出現20通 |
| 発券期限 | `ticketingDeadline` | datetime | 発券忘れ防止。出現34通 |
| 枚数 | `ticketCount` | int | 取得123通。同行者の有無は `ticketCount > 1` で推定 |
| 同行者の有無（派生） | `hasCompanion` | bool（派生） | 枚数から自動推定。将来のB-2(コミュニティ)/B-3(SNS拡散)の布石 |
| 詳細ページURL | `sourceUrl` | string | 公演詳細への導線。LivePocket等で取得可 |

### 見送り
- ゲスト: 出現2通のみで実装コストに見合わない。アーティスト/出演者の枠でLLMが拾えば足りる。

### 既存仕様との関係
- `03_gmail_integration_spec.md` のスキーマには `deadlineDate`/`deadlineTime` が単一で存在するが、本決定で「申込締切・当落発表・入金期限・発券期限」を**種類ごとに分離**する。期限の種類が混在すると正しいリマインドが出せないため。
- `ticketFormat`(paper/digital) は既存スキーマにあり、メール種別（発券系=紙寄り/スマチケ・DL=電子）で判別する。
