# データ分裂の診断・復旧 ＋ 足場（プレビュー≠本番）問題 — 通し指示書

作成: 2026-06-22 ／ 重大度: 高（データ整合・本番表示）。
関連: doc31（方法B・環境分離）／doc32（足場再構築・案X/Y）／#memolog DBドリフト4回。

## 0. 何が起きているか（症状の整理）
- **Manusプレビュー**（`3000-….manus.computer`＝dev server・dev DB）：ライブ履歴は出る（40申込/35当選/34参加）が、**友達が「まだ友達がいません」＝消えている**。
- **本番** `liverico.app`（prod DB）：**ライブ履歴が消えている**。

→ データが **dev DB / prod DB に分裂**している疑いが濃厚。**「消滅」と確定したわけではない**（プレビューに現に出ている＝どこかのDBに実在する）。まず「何がどのDBに居るか」を地図にする。

## 1. 根本原因（単発バグではない・足場の問題）
今日の分裂は doc32 で洗い出した Manus の構造的欠陥の必然の結果：
- **②** `webdev_execute_sql` が本番DB（`DRIZZLE_DATABASE_URL`＝us-east-1）固定 → スキーマ/データ変更が片側だけに入りドリフト（過去4回）。
- **④** 本物のステージングが作れない（1プロジェクト=1URL）→ **プレビューは dev DB・本番は prod DB＝そもそも別物**。

**重要な含意：プレビューと本番が違うDBを見るなら、「本番前に確認する」というプレビューの役割が成立しない。** むしろ「プレビューでOK→本番で別物」の事故装置になる。これは案Y（標準スタックへ移行＝プレビュー＝本番同型のステージングが手に入る）の決め手の実証。ただし判断は後（§4）。

## 2. 【最優先・マナスに今すぐ】データ地図を取る（読み取り専用・絶対に変更しない）

```
【至急・診断のみ・データを一切変更しないこと（読み取り専用）】
症状：
- Manusプレビュー(3000-….manus.computer＝dev DB)：ライブ履歴は出るが「友達」が消えた。
- 本番 liverico.app(prod DB)：ライブ履歴が消えた。
→ データが dev DB / prod DB に分裂している疑い。消滅か接続ズレかを切り分けたい。

絶対厳守：
- INSERT/UPDATE/DELETE/DROP/ALTER を一切実行しない（完全に読み取り専用）。
- 再取込(Gmailバックフィル等)もしない。データを上書き/再生成しうるため。

1. 接続先の棚卸し（ホスト/DB名を明示）：
   - dev/プレビューが読むDB = DATABASE_URL_DEV（ap-northeast-1 /test）
   - 本番が読むDB         = DATABASE_URL（us-east-1）
   - webdev_execute_sql   = DRIZZLE_DATABASE_URL（どこを向くか）
2. 林のuserIdを特定。両方のDB(dev/prod)で件数を読む：
   - SELECT COUNT(*) FROM lives    WHERE userId = <林>;
   - SELECT COUNT(*) FROM friends  WHERE userId = <林>;  ← 友達テーブル名は実体に合わせる
   - 申込/当選/参加カウントの元カラムの件数も両DBで。
3. 友達テーブルの正確な名前・構造を確認（friendships / follows 等の可能性）。
   どのDBに友達レコードが在るかを明示。
4. usersテーブルも両DBで確認：林のuserId/publicId が dev/prod で同一か
   （別IDだと「別アカウント扱い」で見えていない可能性）。
5. 直近デプロイで DATABASE_URL（本番接続先）が差し替わっていないか確認。
報告：
  ① 各DBの接続先（dev/prod/SQLツール）
  ② lives と friends の件数を dev DB・prod DB それぞれ
  ③ ライブはどのDB・友達はどのDBに居るか（分裂状況の地図）
  ④ 林のuserId/publicId が dev/prod で一致するか
  ⑤ 「接続ズレ（データは在る）」か「実消失」かの判定
修正は一切しない。この地図が出てから復旧方針を決める。
```

## 3. 復旧方針（地図が出てから・非破壊が鉄則）
地図の結果で分岐：
- **接続ズレ＝データは在る（最有力）**：本物が在るDBから、本番が読むDBへ**検証付きの非破壊コピー（INSERT … SELECT＋ユニーク制約で冪等）**で寄せる。**上書き/全削除はしない。** 件数を前後で照合し、合わなければ止める。
- **userId/publicId不一致が原因**：データは在るが別ID紐付けで見えていない＝**コピー不要**、紐付け（参照）を正す（慎重に・本番は確認後）。
- **実消失が確定した場合のみ**：バックアップ/スナップショットからの復元を検討（破壊操作は明示確認＋事前記録なしに実行しない）。

**非交渉（安全）：**
- 本番DBへの DELETE/DROP/TRUNCATE/広範囲UPDATE は、対象件数を`SELECT COUNT`で確認＋バックアップ（別テーブルへINSERT SELECT等）してから、明示確認の上でのみ。
- 復旧でも再取込（Gmail等）でデータを作り直さない（本物を潰すリスク）。
- 本文非保存・本番を汚さない原則は維持。

## 4. 足場の本決め（データ復旧の後）
今日「プレビューが本番と別物＝確認装置として無価値」を実体験した。これは doc32 の案Y（標準スタックへ移行）の最大の決め手。
- **案X（Manus内クリーン化）**：②④が構造的に直せない＝プレビュー≠本番が残る＝今日の事故が再発しうる。
- **案Y（標準スタックへ移行）**：プレビュー＝本番同型のステージング・本番ログ可視・マイグレーション一本道・テストが揃う＝根治。コードとデータは保持して移行。
- **進め方**：①まずデータ地図と復旧で「今のデータを守る」→②案Yの第一歩＝Manusのコードを棚卸し（GitHubに出して結合度確認）→③目標構成・データ移行手順・切替・開発体制を設計してから着手（big-bang禁止・並行構築・ロールバック可）。
- ※案Yは林が検討中＝決定保留。今日の実証を判断材料に本決めする。
