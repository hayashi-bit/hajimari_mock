# モニター向け連絡文 ＋ フィードバック記録の仕様（admin/feedback）

作成: 2026-06-21 ／ Gmail取り込みのテスト運用。テスト方法の本体は `19_gmail_account_scope_spec.md`。

---

## Part 1. モニター（テストユーザー）にどう動いてほしいか

### 前提（共有しておくこと）
- このアプリのコアは「**現場好き＝マイナー/インディーのライブに通う人**」。だから**小さなバンドのチケットメールまで自動で拾えるか**が価値。モニターは「**取りこぼしを一緒に見つけてくれる開発パートナー**」。
- **プライバシー：メール本文はサーバに保存していません。**読み取りは「チケット情報の抽出」だけに使い、本文は残しません。安心して連携してOK。
- **1人＝自分のGmail1本だけ**を連携。他人のメールは取り込まれないし、他人からも見えません。

### 連絡文（そのまま送れる・コピー用）
```
【ライブリコ モニター協力のお願い】

いつもありがとうございます。Gmailからチケットメールを自動で取り込む機能を
テスト中です。あなたの「参戦履歴」が手入力ゼロで復元できるかを見ています。
小さなライブ・インディーのチケットまで拾えるようにしたいので、ぜひ協力してください。

■ やってほしいこと
1. アプリの「Gmail連携」から、ご自身のGmailを連携してください。
   （事前にあなたのGmailを許可リストに登録済みです。エラーが出たら教えてください）
2. 「チケットメールを取得」を押し、表示されたプレビューを確認してください。
3. 中身が合っているか見てください（公演名・日付・会場・枚数・ステータス）。

■ 見つけたら教えてほしいこと（フィードバック画面から）
A. 取り込まれていないチケットメールがある
   → そのメールの「差出人アドレス（または会社名）」と「件名」を教えてください。
B. チケットじゃないのに取り込まれている（宣伝・スポーツ・通知など）
   → 同じく「差出人」と「件名」を教えてください。

■ お願い
・本文の中身（個人情報）は送らなくて大丈夫です。差出人と件名だけで十分です。
・プライバシー：メール本文はサーバに保存していません。抽出だけに使っています。

小さなバンドが1つでも多く拾えるよう、気づいたことは何でも教えてください！
```

### 補足（運用メモ）
- 連携でエラー（403 access_denied）が出る人は、テストユーザー未登録の可能性。`docs/livelog/19` の手順でGmailを追加する。
- モニターには「**抽出の正しさはプレビューで本人が確認**」してもらう（こちらは本文を見ないため）。

## Part 2. フィードバック記録（admin/feedback）の仕様

URL（Manus側）: https://livelogapp-nhdbwh2e.manus.space/admin/feedback
※ クロコ環境からは開けないため、ここでは「何を・どう記録させるか」の仕様として定義。実装はManus。

### 目的
モニターからの「取り込まれない／誤取込」報告を**構造化して貯め**、①網漏れ/②解析漏れの切り分け→クエリ追加 or fixture追加→回帰、のチューニングに回す。

### 記録するフィールド（本文は保存しない＝プライバシー厳守）
| 項目 | 必須 | 内容 |
|---|---|---|
| 報告種別 | ✅ | `取り込まれない(miss)` / `誤って取り込まれた(false_positive)` |
| 差出人（アドレス or ドメイン or 会社名） | ✅ | 例：`members_info@pia.co.jp` / `ぴあ` |
| 件名 | ✅ | 例：`[重要]【チケットぴあ/PIA】未発券チケットお引き取り…` |
| 公演名（分かれば） | 任意 | 例：`ずっと真夜中でいいのに。` |
| 自由メモ | 任意 | 補足（ベンダー名・状況など） |
| 報告者ユーザーID | 自動 | ログインユーザー |
| 受信日時/報告日時 | 自動 | 並べ替え・突合用 |
| 解析ステージ判定 | 後追い | `網漏れ` / `解析漏れ` / `仕様どおり除外` を運用側で付与 |
| 対応ステータス | 後追い | `未対応` / `クエリ追加` / `fixture追加` / `完了` |

> ⚠️ **保存してはいけないもの＝メール本文・チケット番号・URL等の中身**。集めるのは差出人＋件名＋（任意で公演名）の最小メタデータのみ。

### 画面UI（最小）
- 一覧：種別・差出人・件名・報告日時・対応ステータスでフィルタ/ソート。
- 詳細：上記フィールド＋運用側メモ（ステージ判定・対応ステータスを更新）。
- ユーザー側の入力導線：プレビュー画面に「このメールが取り込まれない／これは違う」を報告するボタン（差出人・件名を自動でプリフィルできるとベスト。本文は載せない）。

### Manusへの指示（そのまま渡す）
```
【admin/feedback の記録仕様】
1. モニター報告を構造化保存。フィールド：
   報告種別(miss/false_positive・必須)、差出人(アドレスorドメインor社名・必須)、
   件名(必須)、公演名(任意)、自由メモ(任意)、報告者ID(自動)、報告日時(自動)、
   解析ステージ判定(網漏れ/解析漏れ/仕様除外・運用後付与)、対応ステータス(未対応/クエリ追加/fixture追加/完了)。
2. プライバシー厳守：メール本文・チケット番号・URL等の「中身」は保存しないこと。
   集めるのは差出人＋件名＋(任意)公演名の最小メタデータのみ。
3. 一覧は種別・対応ステータスでフィルタ/ソート可能に。
4. ユーザー導線：取り込みプレビューに「取り込まれない/これは違う」報告ボタンを置き、
   差出人・件名をプリフィル（本文は載せない）。
```
