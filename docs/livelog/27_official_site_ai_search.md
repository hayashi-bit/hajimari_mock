# 公式サイト AI検索（外れ対策＝実Web検索グラウンディング）仕様＋Manus指示

作成: 2026-06-22 ／ 目的＝ライブの「公式サイト」欄のAI検索が外れる問題を、構造から直す。
関連: コンセプト（予習・発見）／CLAUDE.md ルール3（実参戦リストで検証・前髪ぱっつん少年が試金石）。

## 1. なぜ外れるか（根本原因）
現状のAI検索は、**LLM（大規模言語モデル）が実際に検索せず“記憶からURLを推測”**している疑いが濃い。
- LLMは、それっぽいが**存在しない/誤ったURLを創作**する（ハルシネーション＝幻覚）。
- とくに**マイナー/インディーはモデルの記憶に無い**ため、コアユーザー（前髪ぱっつん少年系）でほぼ全滅。

→ 「クエリ（検索語）の工夫」だけでは不足。**実在の検索結果に当てて、その中から選ばせる**設計に変える。

## 2. 直し方（アーキテクチャ）＝検索→選別→検証
```
[1] クエリ生成（構造化フィールドから）
      ↓
[2] 本物のWeb検索API（実在URL＋タイトル＋スニペットを取得）
      ↓
[3] LLMが検索結果の中“だけ”から公式情報源を選ぶ（URL創作禁止・優先順位つき・確信度つき）
      ↓
[4] URL生存確認（HTTP 200・パーキング/404でない）
      ↓
[5] 確信度が高ければ自動保存、曖昧なら候補3〜5件を提示し人が1クリック確定（A＋B）
```
**鉄則：LLMは検索結果に無いURLを出力してはならない。** 出せるのは「検索結果に実在したURL」だけ。

## 3. クエリ設計（狙いで分ける）
【確定（2026-06-22）】狙いは **① アーティストの公式HP**。ただし**その中でもライブ/スケジュール情報が載っているサイトを優先**する（林さんの意図）。公演ごとの特設ページが第一目的ではない。
- **既定＝アーティスト公式HP**：`{artistName} 公式サイト` / `{artistName} オフィシャル` / `{artistName} official site`。
  - ⚠️ **日付・会場は入れない**（公式HPは日付非依存。入れると外れる）。
  - 一般名で曖昧なときだけ `{artistName} アーティスト` / `バンド` / `ライブ` で曖昧さ回避。
  - **同点なら、snippet に LIVE/SCHEDULE/TOUR/ライブ/スケジュール/公演 を含む公式HPを優先**（ライブ情報が載っている公式を上位に）。
- **公演ページ狙い（任意モード・将来）**：`{artistName} {venue} {YYYY-MM-DD} 公演` / `{artistName} {tourTitle}`。今回の既定ではない。
- **既存データを種に使う**：レコードに X(旧Twitter) URL・チケットページがあれば**強い手がかり**として併用。

## 4. マイナー対応の優先順位（フォールバック＝代替の連鎖）
インディーは公式HPが無いことが多い。1個に賭けず順に当てる：
**① 公式HP → ② X(旧Twitter)公式アカウント → ③ レーベル/事務所の該当ページ → ④ ライブハウス/イベントの公演ページ。**
公式HPが無ければ X を返すだけでも、予習・発見に実用的。

## 5. LLM プロンプト（選別役・そのまま実装可）
### System
```
あなたは「ライブ情報の公式サイト特定アシスタント」。
与えられた“実在するWeb検索結果”（url・title・snippet）の中から、
指定アーティストの公式情報源を選ぶ。

厳守:
- 検索結果に無いURLは絶対に出力しない（URLを創作・推測しない）。
- 判断できなければ best は null。無理に選ばない。
- 優先順位: ①アーティスト公式HP ②X(旧Twitter)公式アカウント
  ③所属レーベル/事務所の該当ページ ④ライブハウス/イベントの公演ページ。
- ①の公式HPが複数候補なら、snippet に LIVE/SCHEDULE/TOUR/ライブ/スケジュール/公演 を
  含む“ライブ情報が載っている公式HP”を上位にする。
- 同名の別人・非公式まとめ・チケット転売・歌詞サイト・ECは公式ではない（除外）。
- 各候補に type と confidence(0..1) と reason(日本語・短く) を付ける。

出力は次のJSONのみ:
{
  "best": { "url": "...", "type": "official_hp|x|label|event|other", "confidence": 0.0, "reason": "..." } | null,
  "candidates": [ { "url": "...", "type": "...", "confidence": 0.0, "reason": "..." } ]
}
```
### User（テンプレート）
```
アーティスト: {{artistName}}
（任意）公演名: {{title}} / 会場: {{venue}} / 日付: {{liveDate}}
（任意）既知の手がかり: {{knownXUrl}} {{knownTicketUrl}}

検索結果:
{{#each results}}
- url: {{this.url}}
  title: {{this.title}}
  snippet: {{this.snippet}}
{{/each}}
```

## 6. 自動保存 / 候補提示のしきい値
- `best != null` かつ `confidence >= 0.8` かつ **URL生存確認OK** → 自動保存（ユーザーは後から修正可）。
- それ未満 / 生存確認NG → **候補3〜5件をUIに提示**して1クリック確定（誤登録を防ぐ）。
- 全滅なら「見つからない・手動入力」を促す（現状の空表示を維持）。

## 7. 検索APIの候補（採否は実リストで検証）
本物のWeb検索が必須。主要候補を漏れなく：
- **Tavily**（LLM向け検索・グラウンディングに最適化）／**Brave Search API**（独自インデックス・安価）／**Google Programmable Search(CSE)**（網羅性・公式）。
- ⚠️ **採否は机上で決めない**。CLAUDE.mdルール3に従い、**林の実参戦リスト（マイナー含む）でヒット率を検証**してから決める（前髪ぱっつん少年が取れるか）。比較表を残すこと。

```
【公式サイトAI検索の作り直し：実Web検索グラウンディング＋候補提示】

現状のAI検索はLLMがURLを“推測”していて外れる（特にマイナー）。下記に作り直してほしい。

■ アーキテクチャ
1) ライブのフィールド（artistName・任意でtitle/venue/liveDate・既知のX/チケットURL）から検索クエリを生成。
   ・既定はアーティスト公式HP狙い: 「{artistName} 公式サイト / オフィシャル / official site」。日付・会場は入れない。
   ・曖昧な一般名のみ「アーティスト/バンド/ライブ」を足して曖昧さ回避。
2) 本物のWeb検索API（Tavily / Brave / Google CSE のいずれか）で実在のurl/title/snippetを取得。
3) LLMは「検索結果の中だけ」から公式情報源を選ぶ。URL創作は禁止。判断不能はnull。
   優先順位: ①公式HP ②X公式 ③レーベル/事務所 ④ライブハウス/イベントページ。

4) best.url の生存確認（HTTP 200・パーキング/404でない）。
5) confidence>=0.8 かつ 生存OK なら自動保存。未満/NG は候補3〜5件をUI提示→1クリック確定。全滅は手動入力を促す。

■ 検証（採用前に必須）
・林の実参戦リスト（マイナー含む）でヒット率を実測し、比較表を出す（机上で決めない・前髪ぱっつん少年が試金石）。
・検索APIは主要候補（Tavily/Brave/Google CSE）を並べ、カバー率＋コストで比較。

■ プライバシー/コスト
・本文等のPIIは検索に流さない。LLM/検索の呼び出し回数は1保存あたり最小に。
```
