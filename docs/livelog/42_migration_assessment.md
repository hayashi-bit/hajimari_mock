# 移行アセスメント：Manus→Claude Code（miiiroチャット回答＋総括）

作成: 2026-06-25 ／ 目的＝ライブリコの開発基盤をManusから標準スタック+Claude Codeへ移すかの判断材料を確定。
関連: doc31(方法B)/32(足場再構築・案X/Y)/13系handover。出所: miiiro(過去にManus→Claude定着)チャットの回答。

## 1. miiiroチャット回答の要点
- **一番重かったのは配線（デプロイ設定）。アプリ品質ではない。** tsconfig修正は数分、Vercelが
  デプロイしない原因究明に3時間（VercelのGitHub AppとClaude Code GitHub Appが別物）。
- 「定着はしたが重かった」＝**デプロイが通るまで“何をやっても本番に反映されない”ため、バグか設定か
  コードか切り分け不能な状態が長く続く**。構成が複雑なほどこのループは伸びる。
- **段取り（配線疎通→データ移行→機能移行→切替）は正しい。** ただしライブリコはmiiiroと別次元。

## 2. もう一度やるなら最初に固める（初日の規律）
1. **Hello Worldを先に本番まで通す**（機能を書く前に空ページが本番ドメインで出る）。
2. **ローカルDBで完結（dev/prod分離）**＝docker-compose。開発中は本番DBに触れない（ドリフト4回の根治）。
3. **.env.local / .env.production を明示分割**＝どの変数がどの環境かを最初に図解。
4. **本番エラーログが見える状態を先に作る**（Sentry or ホストの標準ログ）＝目隠しデバッグを最初に潰す。
5. **mainブランチ保護（直push禁止・PR必須）**。

## 3. ライブリコ規模の沼化リスク（高い順）
| # | リスク | 内容 | 対策 |
|---|---|---|---|
| 1 | **Manus Auth（最大）** | 認証は全ページ・全APIに絡む。剥がして差替（Clerk/Supabase Auth/Auth.js）でOAuthコールバック・セッション・userId参照が全部変わる。「配線だけ」のつもりが認証再実装で沼る | 認証移行を**単独の最初のタスク**にして他より先に完結 |
| 2 | forgeストレージ（中） | 既存画像が移行先に無い状態で疎通すると画像全崩れ。モニターに見せると体験崩壊 | 画像移行のタイミングを計画に明示・疎通はユーザーに見せない段階で |
| 3 | ホスティング選定（中） | Expressサーバーありでサーバーレス(Vercel)に不向き（コールドスタート/タイムアウト/WS不可）。Railway/Render/Fly向き。フロントVercel+バックRailwayの分割も現実的。**選定を最初に間違えると配線やり直し** | 最初にホスティングを確定 |

## 4. 重要な設計指針（沼回避）
- **「配線疎通」の定義を絞る**：Manus Authなし・画像なしでも動く**最小構成が本番で動く**まで。
- **認証とストレージは「配線フェーズ」ではなく「機能移行フェーズの先頭」**に置く（配線に混ぜない）。

## 5. コスト見積もり
| 項目 | miiiro | ライブリコ |
|---|---|---|
| 配線難易度 | 低(静的+1API) | 高(Auth/ストレージ/DB全部絡む) |
| 沼化リスク | デプロイ設定のみ | 認証・ストレージ・ホスティング選定 |
| Hello World疎通 | 簡単 | 中(認証なしで動かない部分あり) |
| 移行期間 | 数日 | **最低2〜3週間** |

## 6. 総括（クロコの判断）
- 移行は**やる価値がある／今が窓（無料モニター・データ小）／成功前例(miiiro)あり**。
- ただし**2〜3週間の本気プロジェクト**。片手間では沼る。
- **最終GOの最後のピース＝Manus Authの結合の深さ**。コードを読むまで「2〜3週間か、もっとか」は確定不能。
  → **次の一手は棚卸し（ManusのコードをGitHubへpush→クロコが結合度を読む）**。

## 7. 実務分岐（今すぐ効く）
- **doc41の重い機能（プロフィール刷新・公開設定・バッジ・座席共有＝バック重心）はマナスに丸投げしない。**
  移行するなら「後で移すコード」が増える＋クレジット燃焼。
- **マナスでは“モニターが今困るP0バグだけ”**直す（採用ボタン/カレンダー/ピンク輪っか/メール重複統合）。
- doc41の重い機能は**新スタックで作る前提でホールド**。
- 移行中もマナスは動かしたまま＝**モニター2人の体験を止めない（並行構築・無停止・ロールバック可）**。

## 8. データ安全（全移行・ヘビーユーザー実データ）
- 移行前に**検証付きフルバックアップ（全テーブルdump＋件数記録）＝step0**。バックアップ協力は依頼可。
- データ移行は件数・整合をチェックしてから切替。切替はロールバック可能に。
- liverico.appドメイン・モニターアクセス・本文非保存原則は維持。

## 9. 要決定（林）
- **移行を本決めするか（2〜3週間コミット）。** 本決めなら次は棚卸し→Auth結合の実測→目標スタック確定。
