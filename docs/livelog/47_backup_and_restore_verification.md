# 移行前 バックアップ＆復元検証（Manus指示）

作成: 2026-06-26 ／ 目的＝移行に着手する前に、本番データを「完璧に復元できる」状態を確証する。
モニター（ヘビーユーザー2人＋林）のデータを万が一からも守る。関連: doc44/46移行計画。

## 大原則
- **本番DBを一切変更しない**（エクスポート＝読み取りのみ。INSERT/UPDATE/DELETE/DROP/ALTER禁止）。
- 復元検証は**本番でもdevでもない“検証用の別DB”**に対して行う（本番に絶対流さない）。
- 秘密（接続文字列・鍵）は出力に貼らない。本文非保存の原則は維持（DBに本文は無い前提だが確認）。
- 成果物は**林がManusの外にダウンロード/保管できる形**で出す（移行でManusを離れるため）。

---

## マナスへの指示（全文）

```
【最優先・移行前の安全策：本番バックアップ＋“完璧に復元できる”検証】
移行に着手する前に、本番データを完全にバックアップし、それが本当に元通り復元できることを
検証してほしい。本番DBは一切変更しないこと（エクスポートのみ）。

■対象
- 本番DB（DATABASE_URL = us-east-1 / nHdBwH2e...）= 最重要
- 開発DB（DATABASE_URL_DEV = ap-northeast-1 / test）= 念のため
- 全17テーブル（users, lives, ticket_images, artists, setlist_items, memos,
  import_tokens, gmail_tokens, gmail_analysis_cache, feedbacks, gmail_email_decisions,
  friendships, gmail_sync_jobs, gmail_sync_results, live_performers, follows ほか実在する全部）

■STEP1. フルダンプ（読み取りのみ）
- 本番DBの全テーブルを mysqldump 相当でダンプ（スキーマDDL＋全行データ）。
  ・文字コード・タイムゾーン・外部キー順序が壊れない形式で。
  ・1ファイル（例 liverico_prod_YYYYMMDD.sql）にまとめる。
- 開発DBも同様に別ファイルでダンプ。

■STEP2. 件数マニフェスト（後で照合する台帳）
- 各テーブルの行数を記録：SELECT COUNT(*) を全テーブル分。
- 主要テーブルは追加で整合用の指紋も記録：
  ・users / lives / friendships / live_performers / artists は CHECKSUM TABLE の値、
    または MAX(id) と主要カラムの件数（例 lives は userId別件数）。
- ダンプファイルの sha256 とファイルサイズも記録（破損検知用）。
- これらを manifest（表）にまとめる。

■STEP3. ★復元検証（「完璧に復活できる」の証明・ここが本丸）
- 本番でもdevでもない【検証用の新しい空DB】を1つ用意（例 liverico_restore_test）。
  ※本番/devには絶対に流さない。検証用は隔離。
- STEP1のダンプ（本番分）をその検証用DBへリストア（インポート）。
- リストア後、ソース(本番)と復元先で次が完全一致するか照合：
  ・全テーブルの行数（COUNT）が1件残らず一致するか
  ・STEP2で取った CHECKSUM TABLE / 指紋が一致するか
  ・抜き取り確認：林・ヘビーユーザー2人の users 行、その lives の代表数件、
    friendships が、復元先でも同一内容で存在するか
- 1つでも不一致があれば「復元不可」として原因を報告（ダンプ形式・文字コード・FK順序等）。
- 検証用DBは結果報告後に破棄してよい（本番には影響しない）。

■STEP4. 成果物（林がManusの外に保管できる形で）
- ダンプファイル（本番・dev）のダウンロードリンク/保存先。
- manifest（テーブル別件数・CHECKSUM・sha256・サイズ・取得日時UTC）。
- 復元検証の結果：全テーブル「一致/不一致」の一覧と総合判定（PASS/FAIL）。
- 「このダンプから完全復元できることを確認した」or「ここが不一致＝対策必要」を明記。

■絶対にやらないこと
- 本番DBへの書き込み（INSERT/UPDATE/DELETE/DROP/ALTER）。
- 検証データを本番/devへ流すこと。
- 秘密情報（接続文字列・APIキー・.p8）を出力に貼ること。

■完了報告
- PASS（完全復元を確認）か FAIL（要対策）か。FAILなら原因と次の手。
- 林がダウンロードすべきファイルと、その保管手順。
```

## 補足（クロコメモ）
- これがPASSして初めて移行のデータ移行/切替に安心して進める（データ移動ゼロのTiDB保持でも、保険として必須）。
- 林はダンプを**自分のローカル/別ストレージにも保管**（Manusの外）。移行でManusを離れるため。
- 移行直前（切替の当日）にも、最新のダンプをもう一度取り直す（差分を最小化）。
