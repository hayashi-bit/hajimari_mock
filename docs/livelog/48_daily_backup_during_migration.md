# 移行期間中の日次バックアップ（Manus指示）

作成: 2026-06-26 ／ 目的＝移行が終わるまでの数日、毎日バックアップを取り、いつでも最新で戻せる状態を保つ。
関連: doc47(初回フルバックアップ＋復元検証=PASS)／doc44/46移行計画。

## 設計（無駄なく安全に）
- 初回(doc47)で「ダンプから完全復元できる」ことは検証済み＝手順は信頼できる。
- よって日次は軽く：**ダンプ＋件数マニフェスト＋sha256**を毎日。
- **フル復元検証は“切替の直前”にもう一度**だけ実施（毎日はやらない＝重いので）。
- 保管：直近7日分を保持。林が毎日Manusの外へダウンロード。

## マナスへの指示（全文）

```
【移行期間中の日次バックアップ・自動化】
移行が終わるまでの間、本番データの日次バックアップを取り続けてほしい。
本番DBは読み取りのみ（変更しない）。doc47の初回フル検証はPASS済みなので、
日次は「ダンプ＋件数マニフェスト＋sha256」を毎日取る軽量版でよい。

■頻度・自動化
- 1日1回、決まった時刻に自動実行（cron/Heartbeat等）。
  ※自動化が不安定なら、毎日決まった時刻に手動実行でも可（その場合は手順を残す）。

■毎日やること（本番DB・読み取りのみ）
1. 本番DB(DATABASE_URL=us-east-1)の全17テーブルをフルダンプ（mysqldump相当）。
   ファイル名に日時(UTC)を入れる。例 liverico_prod_YYYYMMDDTHHMMSSZ.sql
2. 各テーブルの行数(COUNT)とMAX(id)を manifest_YYYYMMDD.json に記録。
3. ダンプの sha256 とサイズを記録。
4. その日のファイルを zip にまとめ、林がダウンロードできる場所に出力。
5. 直近7日分を保持し、それより古いものは削除（容量管理）。

■毎日の完了通知（林が確認できる形で）
- 取得日時(UTC)、本番の総行数、主要テーブル件数(users/lives/artists/gmail_*)、
  zipのダウンロードリンク、zipのsha256。
- 前日比で件数が異常に減っていないか（激減があれば警告）。

■切替の直前だけ（移行のCutover当日）
- doc47と同じ【フル復元検証】を最後にもう一度実施（検証用の別DBへ復元し全テーブル一致確認）。
- これがPASSしたら切替に進む。

■絶対にやらないこと
- 本番DBへの書き込み（INSERT/UPDATE/DELETE/DROP/ALTER）。
- 秘密情報（接続文字列・APIキー・.p8）を出力に貼ること。
- 検証データを本番/devへ流すこと。

■完了報告（初回）
- 自動化を設定できたか（できなければ手動運用の手順）。
- 1本目の日次バックアップのダウンロードリンクとmanifest。
```

## 林の運用
- 毎日 出てきたzipを**Manusの外（ローカル/Google Drive等）にダウンロード保管**。
- 移行完了後は、新スタック側で定期バックアップを設定し直す（Manus依存をやめる）。
