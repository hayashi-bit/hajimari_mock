## ① フライヤー（ブローシャー/チラシ）保存・表示・記録

```
【ビジュアル①：フライヤー（公演の告知画像）を保存・表示】

・ライブ詳細に「フライヤー画像」を複数添付できる（既存のフライヤー投稿/storage実装を活用）。
・一覧/カード/詳細でサムネイル表示→タップで拡大。
・公開範囲：公式の宣伝物だが、初期値は「友達まで」が無難（権利配慮・非公式明記と連動）。
・画像はユーザー投稿＝storageProxy経由でよい（ロゴ等の静的アセットとは別扱い）。
・モバイルで撮影/アップロード→表示までを実機確認。
・将来：このフライヤー画像をAI出演者抽出のソースにも使う（今は保存・表示まででよい）。
■ 報告：投稿・表示・公開範囲の動作。
```

### （将来）フライヤー→AI出演者抽出のLLMプロンプト
フライヤーには出演者名が載るため、画像→出演者リスト抽出に使える（doc14②）。Gmailと同じ
Claude構造化出力を流用。System例：
```
あなたはライブ告知フライヤーから出演者を抽出するアシスタント。
画像内のアーティスト名のみを、表記そのままで配列にする。会場名・日付・主催・協賛・
チケット情報・キャッチコピーは含めない。読めない/自信がないものは除外。
出力JSON: { "festivalName": string|null, "artists": [string, ...] }
```

## ② チケット画像（電子/紙・半券・QR）

```
【ビジュアル②：チケット画像を記録（プライバシー最優先）】

・ライブ記録に「チケット画像」を添付できる（①フライヤーとは別フィールド/別ラベル）。
・★個人情報（氏名・予約番号・QR・座席）が写る → デフォルトは「自分のみ（非公開）」。
  公開/友達共有は、ユーザーが明示的にオンにしたときだけ。
・投稿時に「QRや個人情報の共有に注意」の軽い注意文を出す。
・画像はユーザー非公開コンテンツ＝storageProxy経由。
■ 報告：投稿・表示・デフォルト非公開の動作。
```

## ③ Apple Music アー写カバー率検証（採用前の必須・ルール3）

```
【Apple Music アー写カバー率検証：実リストでマイナーが取れるか】

アー写＋予習プレイリストは Apple Music API に確定（Spotify不採用）。実装前に
林の実参戦リスト（マイナー含む）でアー写が取れるかを実測する。前髪ぱっつん少年が試金石。

・ハーネス：tools/apple_music_eval/（run_eval.py / fixtures.py / README.md）。
・認証情報（envで渡す・.p8は秘密）：
  APPLE_TEAM_ID=6ZZDG46JS2
  APPLE_KEY_ID=867859YX46
  APPLE_PRIVATE_KEY_PATH=<AuthKey_867859YX46.p8 のパス>
・実行：pip install pyjwt cryptography requests && python tools/apple_music_eval/run_eval.py
・storefront=jp。各アーティストを catalog 検索し、トップ候補の artwork 有無を採点。
■ 報告：
  - 「アー写OK n/16」の比較表（name/状態/マッチ名/アー写URL）。
  - 前髪ぱっつん少年が取れたか（最重視）。同名誤マッチ（例：水槽/アザミ）が無いか。
  - 取れない稀なケースはユーザー投稿で補う方針でよいか判断材料を添える。
・良ければ続けてアー写表示＋予習プレイリスト（出演者の代表曲）の実装へ。
```
