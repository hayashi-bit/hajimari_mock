作成: 2026-06-21 ／ リリースに向け「実サービスに即した必要機能」を確定線で整理。

## 0. リリースの定義（これに必要なものだけ入れる）
- **リリース＝「有料プレセールで“課金が実際に動くか”を検証できる状態」**（04の最優先リスク＝課金実証・チャネル特定）。
- 前提：OAuthテストモード（最大100テストユーザー＝7月有料100人ゴールと一致）、本番URL `liverico.app`。
- 判定軸（ローンチ線に入れるか）：**①無いとプレセールが回らない ②法務/運用上ないと公開できない** ——この2つだけ。それ以外は全部ローンチ後。

## 2. 運営・管理（admin）— ローンチ運用に必要な分だけ
> ここが今まで手薄。実サービスを「回す」ために最小限を定義する。

| 管理機能 | 要否 | 最小仕様（ローンチ版） |
|---|---|---|
| **ユーザー管理画面（一覧）** | 必須 | 登録日／プラン（無料・②・③）／最終ログイン／Gmail連携有無／記録数。検索・並び替え |
| ユーザー詳細・操作 | 必須(軽) | プラン確認・アカウント無効化（サポート/不正対応）。プラン変更は当面Stripe側でも可 |
| **フィードバック管理（admin/feedback）** | 必須 | doc20仕様：種別(miss/false_positive)・差出人・件名・対応ステータス。**本文は保存しない** |
| 課金管理 | 必須(軽) | ローンチ初期は**Stripeダッシュボード流用**で可（課金状況・解約・返金）。自前画面は後 |
| 運用メトリクス | 推奨 | 登録数・有料数・MRR・Gmail取込実績の素朴な集計。最初は手動集計でも可 |
| **一斉メール配信** | ローンチ後寄り | B-1。最初の数十人なら **Resendで手動/軽量送信＋オプトアウト**で十分。セグメント/履歴等のフル機能は後 |
| モニター管理 | 必須(軽) | テストユーザー登録状況の把握、QUOカード配布の控え（スプレッドシートで可） |

**一斉メールの判断（推奨＋理由）**：推奨＝**ローンチ版は「軽量配信＋オプトアウト」だけ**。理由＝最初の100人規模では凝った配信基盤の価値が出ない。ただし**マーケメールを送るならオプトアウト導線は法務上ローンチ時から必須**。フル機能（セグメント/テンプレ/送信履歴）はユーザーが増えてから（B-1本実装）。

## 4. 依存関係（順番の理由）
1. **ログイン解消**（今）→ これが通らないと何も検証できない。
2. **リブランド＋LP一体化＋モバイルP1**（公開できる見た目に）。
3. **Gmail取込の本番デプロイ＋E2E**（③の価値を実機で）。
4. **課金(Stripe)＋プラポリ/規約**（プレセールの器）。
5. **admin最小（ユーザー一覧＋feedback）**（回す土台）。
6. → **プレセールで課金実証**（04の本丸）。

> フリーミアム上限・一斉メールフル・未来ニュースAIは、いずれも「課金が動く」を確認した後。順番を守れば二度手間が出ない。
