LiveLog — ライブ参加記録・管理アプリ

ユーザーが参加したライブ・コンサートの記録・管理・共有を簡単にするアプリ。e+連携、AI解析、セットリスト記録、友達機能などを提供。

lives ルーター

ライブのCRUD・ステータス管理・AI解析・バックアップを担当するコアルーター。

lives.listpublicquery
ユーザーのライブ一覧を取得。statusフィルターを任意指定可能。
input?: { status?: string } // 'pre_sale' | 'lottery_won' | 'attended' | ...
lives.createprotectedmutation
新規ライブを作成する。
input: { title: string artistName?: string liveDate: Date status?: string // default: 'pre_sale' ticketFormat?: 'paper' | 'digital' venue?: string prefecture?: string price?: number seatInfo?: string }
lives.updateprotectedmutation
ライブ情報を部分更新する。id以外は全て任意。
input: { id: string title?: string artistName?: string liveDate?: Date status?: string isConsidering?: boolean ticketFormat?: 'paper' | 'digital' deadlineDate?: Date deadlineTime?: string openTime?: string startTime?: string venue?: string prefecture?: string price?: number seatInfo?: string }
lives.deleteprotectedmutation
input: string // liveId
lives.getByIdpublicquery
ライブ詳細を取得。アーティスト・セットリスト・メモを含む。
input: string // liveId
lives.updateStatusprotectedmutation
input: { id: string, status: string }
lives.suggestTimesprotectedmutation
LLMでライブの開場・開演時間を提案する。
input: string // liveId output: { openTime: string, startTime: string }
lives.parseEplusTextprotectedmutation
e+ページのテキストをLLMで解析してライブ情報配列を返す。
input: string // e+テキスト output: [{ title, artistName, liveDate, venue, ... }]
lives.importParsedLivesprotectedmutation
parseEplusTextの解析結果をまとめてインポートする。
input: [{ title, artistName?, liveDate, venue?, ... }]
lives.importFromBookmarkletprotectedmutation
ブックマークレットから取り込む。
input: { title, artistName?, liveDate, venue?, ... }
lives.parseImageLiveprotectedmutation
画像URLからLLMでライブ情報を抽出する。
input: { imageUrl: string } output: { title, artistName, liveDate, venue, ... }
lives.exportFullBackupprotectedquery
全ライブデータをCSV形式でエクスポートする。
output: string // CSV文字列
lives.importFullBackupprotectedmutation
CSVファイルをインポート。重複チェック・日付正規化処理を含む。
input: { csv: string } output: { success: number, error: number }

事業戦略・ロードマップ
2026-06

事業の核

「複数現場を追うライブ好きが、チケット期限を一度も逃さず、行ったライブを自動で記録できる」。製品は完成済み。今の課題は機能ではなく流通（ユーザー獲得）と課金の実証。

当面の目標

2026年7月：有料顧客 100人

課金形態は買い切り（¥2,000前後）を主軸。送客・チケットサイト買取は規模が出るまで棚上げ。

開発形態

Web + PWA化（ホーム画面アイコン・プッシュ通知）。ネイティブは後でラップ可能。iOSのApple課金強制を避け、Stripeの利益率を守るためWeb継続が合理的。

マネタイズ戦略
課金モデル（near-term の結論）

主軸 買い切り（単体買取）¥2,000前後

churnを気にせず「売れた数」を積める。7月100人スプリントに最も素直。月額/年額サブスクは継続管理が必要で不利。

棚上げ 送客マージン / チケットサイト買取

現状の知名度・利用者数ではB2B交渉が成立しない。会員が伸びて送客できる規模になれば立場・戦略は大きく変わる → その時点で再検討。

出口シナリオ

(A) 自社運営で買い切り販売 → 利益化（現在の主軸）

(B) 事業売却：想定以上に伸びた場合のupside。有料100人等の実績が買い手向けの価値になる。

最優先で潰すべきリスク（機能ではない）
1
課金の実証
プレセールで実際に金が動くか検証。LPに価格を出し先行N名で事前決済。機能の実証 ≠ 課金の実証。
2
最初の100人のチャネル
未定だと「100人」は願望。広告で自然流入は今の規模では起きない。最初の数十人は手売り前提。
ランニングコスト試算（5,000ユーザー/月・クロード×マナス）

2つの独立試算が「LLMが支配的・固定費は小さい・1人あたり月¥15〜25」で一致。手動のみなら約¥5。

サーバー 0〜5,000円（Manus込みでほぼ0）
DB(TiDB) 3,000〜7,500円
ストレージ 450〜900円
メール(Resend) 3,000円
LLM API 33,000円・最大変動要因
合計 約39,000〜50,000円/月（40%稼働）

損益分岐 = 月¥50,000 ÷ ¥2,000 = 月25人の販売で黒字。

買い切りリスクと統合結論（2階建てハイブリッド）

技術顧問・門倉さんの指摘 ①永久に原価が出る ②毎月売り続けないと収益ゼロ → どちらも「継続コストのある機能を永久販売に含めるな」に集約。機能をコスト性質で2階建てにする。

土台

無料 / 買い切り。永久提供しても赤字にならない
課金

仕様確定

Manus OAuth → Google OAuth

Manusアカウント必須をやめ、Googleアカウントでログイン。無料・データ保持・マルチデバイス対応。後続のGmail連携と同じGoogle基盤を使う。

openId 残す・移行照合用
email 移行の照合キー

既存ユーザーのデータ引き継ぎ

初回Googleログイン時、emailで既存usersを照合 → ヒットすればgoogleIdを追記して引き継ぎ（lives等は内部userId紐付けのため移行不要）。ヒットなしなら新規作成。メールが同じならユーザー操作ゼロで自動移行。

仕様確定

チケットメール自動取り込み（汎用設計）

3大サイトに乗らない小規模イベント（Peatix/ZAIKO/LivePocket/teket等）を販売元を問わず取得。特定サービスにハードコードせず、LLMで汎用解析。Googleログインにgmail.readonlyスコープを追加して実現。

提供価値（2層）
① 新規登録
公演名・日付・会場・アーティスト
価格・紙/電子
小規模イベントの取りこぼし解消
② ステータス自動更新
抽選申込 → lottery_pending
当落 → lottery_won / lost
入金 → payment_pending
発券 → print_pending
購入完了 → ticket_secured
パイプライン

Gmail検索（汎用クエリ）→ 本文取得 → 重複チェック（sourceMessageId）→ LLMで構造化抽出 → 新規登録 or ステータス更新に振り分け。ステータスは前進方向のみ自動更新。

プライバシー / 審査

スコープはreadonlyのみ・本文は保存しない・refreshTokenは暗号化。gmail.readonlyは制限付きスコープのため一般公開にGoogleのセキュリティ審査（CASA含む・数週間）が必要。MVPはテストユーザー運用で先行検証、並行して審査申請。

詳細仕様: docs/livelog/03_gmail_integration_spec.md
