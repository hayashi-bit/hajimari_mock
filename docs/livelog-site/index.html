LiveLog — ライブ参加記録・管理アプリ

ユーザーが参加したライブ・コンサートの記録・管理・共有を簡単にするアプリ。e+連携、AI解析、セットリスト記録、友達機能などを提供。

公開URL：https://livelogapp-nhdbwh2e.manus.space/

LP（紹介ページ）：サービス紹介ランディングページを開く →

ブランド・カラートーン

アプリのカラートーンは Aooo『Rooom』アルバムのアートワークを基調にしている。ポップで遊び心のある、ドリッピング（絵の具垂らし）調の多色パレット。UIの主役はピンク→オレンジのグラデーション＋淡いブルーグレー背景。

呼称ポリシー（アプリ / サービス）

LiveLog はブラウザ／PWAベースだが、利用層（ライブ好き・ほぼスマホ）に最も伝わる「アプリ」を基本呼称とする。事業・対外文脈でのみ「サービス」を使う。「サイト」は受動的に聞こえるため避ける。

対ユーザー（LP・アプリ内・SNS・宣伝） →「アプリ」で統一
対ビジネス（事業説明・顧問・資料） →「サービス」
「サイト」 → 使わない（機能性が伝わらない）

※ PWA化でホーム画面常駐＝実体も「アプリ」。コピーは「ライブ参加の記録・管理アプリ」で統一。

事業戦略・ロードマップ
2026-06

事業の核

「複数現場を追うライブ好きが、チケット期限を一度も逃さず、行ったライブを自動で記録できる」。製品は完成済み。今の課題は機能ではなく流通（ユーザー獲得）と課金の実証。

当面の目標

2026年7月：有料顧客 100人

課金形態は買い切り（¥2,000前後）を主軸。送客・チケットサイト買取は規模が出るまで棚上げ。

Web + PWA化（ホーム画面アイコン・プッシュ通知）。ネイティブは後でラップ可能。iOSのApple課金強制を避け、Stripeの利益率を守るためWeb継続が合理的。

マネタイズ戦略
課金モデル（near-term の結論）

主軸 買い切り（単体買取）¥2,000前後

churnを気にせず「売れた数」を積める。7月100人スプリントに最も素直。月額/年額サブスクは継続管理が必要で不利。

棚上げ 送客マージン / チケットサイト買取

現状の知名度・利用者数ではB2B交渉が成立しない。会員が伸びて送客できる規模になれば立場・戦略は大きく変わる → その時点で再検討。

出口シナリオ

(A) 自社運営で買い切り販売 → 利益化（現在の主軸）

(B) 事業売却：想定以上に伸びた場合のupside。有料100人等の実績が買い手向けの価値になる。

A1. ランニングコスト概算（5,000人・40%稼働）— クロード×マナス一致

2つの独立試算が「LLMが支配的・固定費は小さい・1人あたり月¥15〜25」で一致。手動のみなら約¥5。

サーバー 0〜5,000円（Manus込みでほぼ0）

ストレージ 450〜900円
メール(Resend) 3,000円

合計 約39,000〜50,000円/月
詳細内訳（前提・LLM呼び出し・稼働シナリオ）

前提条件

総ユーザー数 5,000人
月間アクティブ率 40%（2,000人）
1人あたり月間ライブ参加 2〜3本
